# ctxlint v0.2 Handoff — Build System Integration

## Context

ctxlint is an open-source CLI + MCP server for linting AI context files (CLAUDE.md, AGENTS.md, MEMORY.md, etc.). v0.1 ships with checks for: dead file path refs, invalid commands, stale context, token waste, and redundancy.

Domains: ctxlint.com, ctxlint.org. Positioning: complementary to enterprise context tools (Unblocked, Augment Code), serving solo and small-team devs.

## Why v0.2

A "Claude Code as build system" pattern is emerging in the community (see vinny.dev/blog/2026-04-25-claude-code-build-system and the companion repo at github.com/vscarpenter/claude-code-build-system). The pattern treats AI context as a layered, observable system: tight CLAUDE.md, long-form standards, hooks, subagents, slash commands, and memory files. Once the layers exist, they drift independently, and that's exactly the surface ctxlint should cover.

v0.2 expands ctxlint from "lint a single context file" to "lint the whole context build system."

## Scope additions

### 1. Preset templates

New CLI verb: `ctxlint init --preset=<name>`. Scaffolds opinionated starter configs into the current repo.

Initial presets:

- `default` — minimal CLAUDE.md + tasks/lessons.md + tasks/todo.md
- `claude-code-build-system` — full layered setup: 200-line CLAUDE.md, coding-standards.md, tasks/lessons.md, tasks/todo.md, .claude/commands/{qspec,tdd,qcheck}.md stubs, .claude/agents/ stubs, settings.json baseline with deny rules
- `solo-dev` — single CLAUDE.md, no extras
- `team` — settings.json baseline + settings.local.json gitignore guidance + cross-machine conventions

Presets are versioned templates shipped inside the ctxlint package. `--preset` flag is additive — running it on an existing repo prompts before overwriting.

If we adapt patterns from public repos (like vscarpenter's), credit the source in a comment header in the scaffolded files. Goodwill matters in this niche.

### 2. New check modules

Five new modules. All read-only by default; all surfaced through both the CLI and the MCP server.

**hook-validation**
- Parse hook definitions in `settings.json` (Stop, PreToolUse, PostToolUse, etc.).
- Flag heavyweight Stop hooks (full test suites, full builds, anything >5s in typical repos).
- Recommend lighter alternatives (typecheck over full vitest).
- Detect hooks that silently swallow errors (missing `set -e`, no exit code propagation).
- Flag hook scripts that reference files outside the repo or use unbounded glob patterns.

**slash-command-sanity**
- Validate `.claude/commands/*.md` files.
- Check for: unbounded scope ("do everything related to X"), missing acceptance criteria, references to nonexistent files or commands, duplicate commands across user vs project scope.
- Flag commands that haven't been touched in 90+ days and don't appear in CLAUDE.md or recent git history.

**subagent-allowlist-review**
- Parse `.claude/agents/*.md` definitions.
- Verify tool allowlists match the agent's role (e.g. read-only review agents must not have Write/Edit/Bash-write).
- Flag overly broad model + tool combos (Opus + full toolset for a narrow review task).
- Warn if subagent prompt references files that don't exist in the repo.

**permissions-audit**
- Parse `settings.json` permissions blocks.
- Flag missing deny rules for: `rm`, `sudo`, `chmod`, reads of `.env` and `**/secrets/**`.
- Detect bloated allow lists (>100 entries) and suggest pruning.
- Differentiate `settings.json` (team baseline) vs `settings.local.json` (personal allowances) and warn if local overrides creep into the team file.

**memory-layer-drift**
- Cross-check `tasks/lessons.md`, `tasks/todo.md`, and CLAUDE.md for consistency.
- Flag stale "Resuming From Here" sections (todo.md not updated in N days).
- Detect lessons referenced in CLAUDE.md but missing from lessons.md (and vice versa).
- Flag duplicate guidance across layers (same rule stated in CLAUDE.md and coding-standards.md).

### 3. MCP server tool surface

Each new check gets a corresponding MCP tool so Claude Code can invoke ctxlint inline during a session:

- `ctxlint_check_hooks`
- `ctxlint_check_commands`
- `ctxlint_check_subagents`
- `ctxlint_check_permissions`
- `ctxlint_check_memory_drift`
- `ctxlint_init_preset` (writes scaffold files; requires explicit confirmation)
- `ctxlint_check_all` (umbrella that runs every check and returns a single report)

Tool descriptions should be tight enough that Claude Code triggers the right one without prompting (model-readable, not human-readable).

## Out of scope for v0.2

- Cross-repo standards sync (deferred to v0.3)
- Hosted dashboard, audit log retention, team RBAC (paid tier, post-v1)
- Real-time IDE plugin (not in roadmap)

## Roadmap

- **v0.1** — core lint checks (existing spec, in progress)
- **v0.2** — build system integration (this doc)
- **v0.3** — cross-repo sync, team config sharing, optional hosted preset library
- **v1.0** — stable CLI, stable MCP tool contract, public docs

## Implementation notes for Claude Code

- All new check modules must follow the existing v0.1 module interface (assume it already exists; do not redesign).
- New checks must be opt-out by default in `ctxlint.config.json` so existing v0.1 users don't get a flood of new warnings on upgrade.
- Each check module ships with at least one fixture-based test demonstrating a passing repo and a failing repo.
- Preset scaffolds live under `presets/<name>/` in the ctxlint repo. Loading is filesystem-based; no network calls.
- Keep the binary lean — no heavy parsers. Use a single markdown parser already in v0.1; reuse for all new modules.
